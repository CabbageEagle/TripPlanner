# HelloAgents Trip Planner 完整项目介绍

## 1. 项目概述

`HelloAgents Trip Planner` 是一个围绕“智能旅行规划”构建的 Agent 工程项目。它不是单纯的“LLM 生成一段文本”，而是一个把大模型能力、工作流编排、规则排程、外部工具调用、用户偏好记忆、版本化存储和离线评估连接起来的完整系统。

项目目标可以概括为三点：

- 把用户的旅行需求转成结构化、可展示、可编辑的多日行程计划。
- 让计划在生成后具备更强的可执行性，而不是停留在“看起来合理”的文本层面。
- 让系统能从用户历史请求与编辑行为中学习，并通过评估体系持续改进。

从工程定位上看，这个项目属于典型的 `Agent Application`：

- 上层是用户交互界面与 API。
- 中层是以 `LangGraph` 为核心的 Agent 编排层。
- 下层是地图工具、排程器、记忆检索、数据库、评估器等能力模块。

它的核心价值不在单次生成，而在于形成了以下闭环：

- `生成`
- `排程`
- `校验`
- `编辑`
- `记忆沉淀`
- `离线评估`
- `再生成`

## 3. 系统整体架构

整个项目可以拆成 6 个主要层次。

### 3.1 表现层：Frontend

前端负责：

- 收集用户旅行需求
- 展示结构化旅行计划
- 可视化时间线、预算、天气、地图标记
- 支持用户编辑计划并提交保存
- 支持导出图片 / PDF

核心页面：

- `Home.vue`：输入表单页
- `Result.vue`：行程结果页与编辑页

### 3.2 接口层：FastAPI API

API 层提供统一的业务入口，核心接口包括：

- `POST /api/trip/plan`
- `POST /api/trip/plan/stream`
- `GET /api/trip/plans/{plan_id}`
- `PUT /api/trip/plans/{plan_id}`
- `GET /api/map/poi`
- `GET /api/map/weather`
- `POST /api/map/route`
- `GET /api/poi/detail/{poi_id}`
- `GET /api/poi/photo`

这一层不是简单转发，而是负责：

- 请求校验
- 调用 Agent 工作流
- 持久化计划和历史版本
- 检索并注入偏好记忆
- 在编辑保存时触发自动重排

### 3.3 Agent 编排层：LangGraph

这是项目的核心。LangGraph 负责把多个推理节点组织成稳定的工作流，而不是依赖单次 prompt 解决全部问题。

主要节点：

- `search_attractions`
- `query_weather`
- `search_hotels`
- `plan_trip`
- `parse_plan`
- `schedule_plan`
- `verify_plan`
- `fix_plan`
- `error_handler`

这层的意义在于把“大模型规划”拆成多个明确职责的阶段，使系统具备更强的可控性、可调试性和可扩展性。

### 3.4 能力服务层：Services

服务层承担具体能力：

- `amap_service.py`：地图 MCP 调用与结果解析
- `scheduler_service.py`：时间线自动排程
- `memory_service.py`：偏好记忆抽取、存储、检索、摘要
- `judge.py`：LLM-as-Judge 评估
- `unsplash_service.py`：景点图片检索

### 3.5 数据层：Repositories + DB

项目使用数据库保存：

- 当前旅行计划
- 历史计划版本
- 用户记忆条目
- 记忆向量 embedding

这里不是简单存一份 JSON，而是把“生成结果、编辑结果、记忆、版本历史”拆开存储，为后续回溯、学习和评估提供基础。

### 3.6 离线评估层：Eval Pipeline

项目包含一条独立于在线生成链路的离线评估 pipeline：

- 测试样例读取
- 调用真实 Agent 或 mock 计划生成
- 调用 Judge 模块多维打分
- 输出结构化评估报告并归档

这是项目中非常典型、也非常有代表性的 Agent 工程模块。

## 4. 核心业务流程

## 4.1 在线生成流程

用户在首页输入：

- 目的地城市
- 日期范围
- 天数
- 交通方式
- 住宿偏好
- 兴趣标签
- 自由文本需求
- 预算和时间限制

后端收到请求后，流程如下：

1. 检索与当前请求最相关的历史记忆。
2. 通过 `summarize_preferences` 压缩成可注入 prompt 的偏好摘要。
3. 调用 LangGraph Agent 工作流生成结构化行程草案。
4. 对 LLM 输出 JSON 进行解析。
5. 调用排程器补全每个景点的建议访问时间和日程时间线。
6. 对结构和内容进行校验。
7. 必要时进入修复节点。
8. 持久化旅行计划、创建版本记录、写入请求记忆。
9. 返回 `TripPlanResponse` 给前端。

这条链路体现的不是“LLM 输出文本”，而是“LLM 参与一个有状态、有规则、有持久化、有反馈的业务流程”。

## 4.2 流式生成流程

项目还支持 `SSE` 流式接口：

- `POST /api/trip/plan/stream`

这个接口会按阶段推送事件：

- `received`
- `retrieving_memory`
- `planning`
- `persisting`
- `done`

这意味着项目不仅支持最终结果，还支持执行过程的可视化，对真实产品化体验更友好。

## 4.3 编辑保存流程

用户在结果页编辑行程后，不是简单覆盖本地数据，而是进入完整后端链路：

1. 前端调用 `PUT /api/trip/plans/{plan_id}`
2. 后端读取原始请求约束
3. 调用 `schedule_day_plan` 对每一天重新排程
4. 保存为新的计划版本
5. 分析用户修改内容，抽取偏好记忆
6. 返回新的结构化计划

这使得系统具备“编辑后自动重排”和“从编辑中学习”的能力，是本项目区别于普通 AI Demo 的关键点。

## 5. Agent 核心设计

Agent 开发的重点不在于“用了 LangGraph”，而在于如何设计每个节点的职责、输入输出以及模型和规则之间的边界。

## 5.1 为什么使用 LangGraph

如果只用单个 prompt 直接生成整份旅行计划，会出现几个问题：

- 很难控制输出结构
- 很难插入外部工具调用
- 很难在失败后局部修复
- 很难加入排程和校验等后处理

LangGraph 的价值在于：

- 定义明确的节点职责
- 把中间状态保存在 `TripPlannerState`
- 用条件边实现“解析失败重试”和“校验失败修复”
- 让工作流具备更强的工程可控性

## 5.2 状态设计

状态结构 `TripPlannerState` 存放：

- 原始请求 `request`
- 中间数据：景点、天气、酒店信息
- 原始计划 `final_plan_raw`
- 结构化计划 `final_plan`
- 违规项 `violations`
- 排程状态 `schedule_applied / schedule_notes`
- 重试计数 `parse_retry_count / verify_count`
- 当前节点状态 `current_step`

这种状态设计让工作流不是“黑盒调用”，而是“可观察、有阶段语义”的系统。

## 5.3 工作流节点职责

### `search_attractions`

负责根据城市、偏好和旅行天数生成景点候选信息。  
这一步强调的是“候选资源组织”，不是完整行程编排。

### `query_weather`

负责补充天气条件。  
天气不是附属信息，它会影响后续时间安排与行程合理性。

### `search_hotels`

负责生成住宿候选。  
住宿在本项目中除了信息展示，还作为排程起点和上下文的一部分。

### `plan_trip`

这是 LLM 的主规划节点。  
它将景点、天气、酒店、用户需求和历史偏好整合成一份结构化 JSON 行程草案。

### `parse_plan`

负责把模型输出从文本转换为结构化 JSON。  
这里处理了 fenced code block、JSON 片段等常见输出形式。

### `schedule_plan`

这是新增的重要节点。  
它不是让 LLM“猜时间线”，而是调用确定性排程器去补全：

- 景点建议开始时间
- 景点建议结束时间
- 每日 timeline
- 每日总时长
- 每日总成本
- 排程 warnings

### `verify_plan`

对计划做结构和完整性校验，例如：

- 必填字段是否存在
- 天数是否匹配
- 每天景点数量是否合理
- 餐饮信息是否完整
- 景点字段是否齐全

### `fix_plan`

当校验发现可修复问题时，使用新的 prompt 让模型做针对性修正。  
这个节点不是重新生成整份计划，而是做“问题驱动的修复”。

### `error_handler`

当解析或修复流程彻底失败时，系统仍会输出备用方案，避免整条链路中断。

## 6. Scheduler：从“看起来合理”到“可执行”

这是项目里非常重要的一层，也是 Agent 工程实践中的亮点。

## 6.1 为什么需要 Scheduler

纯 LLM 生成的旅行计划通常存在这些问题：

- 景点访问时间可能超出营业时间
- 一天安排过满或过松
- 中午、晚上用餐时段被挤掉
- 景点之间缺少真实交通时间
- 行程时间线无法落地

因此项目采用混合架构：

- `LLM` 负责高层语义规划
- `Scheduler` 负责低层时间执行逻辑

## 6.2 Scheduler 的输入与输出

输入：

- 单日行程 `day`
- 配置 `ScheduleConfig`

输出：

- 补全后的 `day`
- 一组 `warnings`

补全字段包括：

- `timeline`
- `visit_start_time`
- `visit_end_time`
- `total_duration`
- `total_cost`

## 6.3 Scheduler 的主要规则

排程逻辑中包含：

- 根据 `daily_start_time / daily_end_time` 控制日程边界
- 在景点访问前后插入午餐 / 晚餐
- 根据营业时间 `opening_hours` 判断是否可访问
- 利用交通时间估计处理景点间移动
- 时间不足时生成 warning，而不是强制失败

## 6.4 与地图服务的关系

Scheduler 默认可以使用固定 `default_travel_minutes` 作为交通时间估计。  
当配置 `schedule_use_mcp_route = True` 且地图服务可用时，可以进一步调用路线规划工具，得到更接近真实的耗时。

这体现了一个典型的 Agent 工程思想：

- 先保证系统在弱依赖条件下可运行
- 再通过可选外部工具逐步增强精度

## 7. MCP 工具与地图服务

## 7.1 为什么使用 MCP

MCP 的价值是把外部能力以“工具”方式接入 Agent 系统，而不是手写一堆散乱的 API 调用逻辑。  
在这个项目中，地图相关能力是典型的 Agent 外部工具场景。

## 7.2 AmapService 的职责

`AmapService` 封装了高德地图 MCP 调用：

- `search_poi`
- `get_weather`
- `plan_route`
- `geocode`
- `get_poi_detail`

## 7.3 本项目的关键技术点：弱结构化结果解析

MCP / 第三方工具的返回并不总是严格统一，因此项目没有把解析逻辑写死，而是设计了一个相对鲁棒的归一化层：

- `_normalize_mcp_result`
- `_extract_poi_list`
- `_extract_weather_list`
- `_extract_duration_minutes`
- `_extract_distance_km`
- `_find_first_list`
- `_find_first_numeric_by_keys`

这套逻辑的工程意义在于：

- 降低对上游返回格式稳定性的强依赖
- 给后续的排程、前端展示和路线估计提供统一数据接口

## 8. RAG 记忆系统

## 8.1 为什么这个项目需要记忆

旅行规划不是一次性任务。用户在多次请求和多次编辑中会不断暴露出稳定偏好，例如：

- 喜欢自然风光还是历史文化
- 更喜欢轻松节奏还是紧凑节奏
- 对住宿有什么长期偏好
- 对预算和交通方式有什么约束

如果系统每次都“从零开始规划”，这些信息会不断丢失。

## 8.2 记忆的来源

项目有两类主要记忆来源：

### 请求记忆

从原始 `TripRequest` 中抽取，例如：

- 偏好标签
- 预算约束
- 时间窗口
- 交通方式
- 住宿偏好

### 编辑记忆

从用户对计划的修改中抽取，例如：

- 删除某类景点
- 增加某类景点
- 改变住宿偏好
- 调整节奏偏好

编辑记忆比请求记忆更有价值，因为它更接近用户真实反馈。

## 8.3 存储方式

数据库中有两张相关表：

- `MemoryItem`
- `MemoryEmbedding`

`MemoryItem` 存放文本记忆和 metadata，`MemoryEmbedding` 使用 `pgvector` 保存向量。  
这使得系统可以进行向量检索，而不只是简单文本匹配。

## 8.4 检索流程

当新请求进入时：

1. 把请求转成查询文本
2. 生成查询向量
3. 用向量相似度搜索最相关记忆
4. 对记忆结果做摘要压缩
5. 把摘要注入 Agent 规划流程

## 8.5 摘要策略升级

这个项目的记忆摘要不是简单地“取前几条”，而是做了更细的排序策略：

- memory type bonus
- source bonus
- recency decay
- retrieval rank
- conflict group 抑制

这意味着系统会尽量避免把互相冲突的偏好同时塞进 prompt，例如：

- “喜欢轻松节奏”
- “希望多安排景点”

这类策略是非常典型的 `Agent Memory Engineering`。

## 9. 数据持久化与版本化

## 9.1 为什么要版本化

AI 生成内容是可变的，用户编辑后往往希望：

- 当前版本可继续修改
- 历史版本可以回看
- 可以对比“生成版”和“编辑版”

因此项目没有只保留一份当前 JSON，而是采用：

- `TripPlanRecord` 存当前版本
- `TripPlanVersion` 存历史快照

## 9.2 版本化的价值

- 支持回溯用户修改轨迹
- 给评估体系提供不同版本样本
- 给记忆系统提供高质量反馈来源

这一设计让项目从“功能演示”进入“真实系统”的范畴。

## 11. LLM-as-Judge 评估体系

这是本项目另一个非常重要的 Agent 技术亮点。

## 11.1 为什么需要 Judge

Agent 项目很容易陷入一个问题：

- “看起来不错”
- 但没有可度量的质量标准

没有评估体系，就很难回答以下问题：

- 换了 prompt 是否更好？
- 加了 scheduler 是否真的提升了质量？
- 新模型版本是否比旧模型更稳定？

## 11.2 Judge 模块做了什么

`judge.py` 提供统一入口 `judge_trip_plan`，支持两种模式：

- `llm`
- `heuristic`

同时支持：

- `auto` 自动切换
- `strict_llm_judge` 强制要求 LLM 评估

## 11.3 结构化评估输出

Judge 输出不是自由文本评论，而是结构化数据：

- `scores`
- `overall_score`
- `summary`
- `issues`
- `confidence`
- `evaluation_mode`

其中多维 rubric 包括：

- `schedule_quality`
- `budget_match`
- `attraction_diversity`
- `completeness`
- `requirement_fit`

这说明项目把评估做成了“可计算、可存储、可追踪”的工程系统，而不是临时用 prompt 打分。

## 11.4 关键技术点

### 严格结构化输出

Judge 使用 `Pydantic schema + response_format + strict JSON schema`，确保输出可解析、可落库、可比较。

### 专用 Judge 模型

项目支持独立的 `judge_model`，避免用同一个生成模型同时做裁判，减少评估偏置。

### 回退机制

当 LLM 评估失败时，系统会自动切换到 `heuristic` 模式，保证整条评估 pipeline 不会因为模型问题中断。

这是一种非常成熟的 Agent 评估思路：

- LLM 提供主观质量判断
- Heuristic 提供系统可用性的保底

## 11.5 run_eval.py 的作用

`run_eval.py` 是项目的本地评估入口，用于：

- 读取 `test_cases.json`
- 选择真实 Agent 或 mock 作为计划来源
- 调用 Judge 评分
- 生成最新报告
- 归档历史报告

这使项目具备了实验管理能力，可以比较不同时间点、不同配置、不同模型的质量表现。

## 12. 为什么这是一个典型的 Agent 工程项目

如果从“Agent 开发”的视角看，这个项目的重点并不是界面，而是以下几个技术点：

## 12.1 工作流编排而不是单轮 Prompt

系统把旅行规划拆成多个阶段，每个阶段有明确职责和状态转移，而不是依赖一个超级 prompt 一步到位。

## 12.2 大模型与规则系统协作

项目采用的是混合智能架构：

- LLM 负责语义层规划
- Scheduler 负责时间规则落地
- Verify/Fix 负责质量约束闭环

这比单独依赖 LLM 更稳定，也更适合生产化。

## 12.3 工具使用不是点缀，而是业务骨架的一部分

地图、路线、天气、地理编码都不是可有可无的辅助功能，而是排程与可执行性的基础数据来源。

## 12.4 记忆是系统级能力，不是简单历史记录

项目把用户请求和编辑行为转成可检索、可摘要、可注入的记忆，说明它在往真正“长期个性化 Agent”方向发展。

## 12.5 评估是第一类模块

很多项目只有生成，没有评估。  
本项目明确建设了 LLM-as-Judge，这意味着它具备持续优化能力，而不是只能做功能展示。

## 14. 项目最值得强调的技术亮点

如果需要把这个项目对外介绍，最值得强调的点有 6 个：

1. 基于 `LangGraph` 的结构化 Agent 工作流，而不是单轮 prompt 生成。
2. 采用 `LLM + Scheduler + Verify/Fix` 的混合架构，提高结果可执行性。
3. 接入 `MCP` 工具链并实现弱结构化结果的统一解析。
4. 支持用户编辑后的自动重排与版本化存储。
5. 引入 `RAG` 记忆系统，把请求与编辑行为转成长期偏好。
6. 建立 `LLM-as-Judge` 评估体系，实现 Agent 质量闭环。

## 15. 总结

`HelloAgents Trip Planner` 的本质不是一个“旅行页面”，而是一个围绕旅行规划场景搭建的完整 Agent 系统。  
它把以下能力真正串成了一个工程闭环：

- `规划生成`
- `工具调用`
- `规则排程`
- `结构校验`
- `用户编辑`
- `偏好记忆`
- `版本管理`
- `质量评估`

从 Agent 开发视角看，这个项目最重要的意义在于：  
它展示了如何把大模型从“会说话的模型”变成“能在真实业务流程中稳定工作的系统组件”。
