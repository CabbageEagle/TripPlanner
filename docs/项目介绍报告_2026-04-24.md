# HelloAgents Trip Planner 项目介绍报告

生成日期：2026-04-24

## 1. 项目概述

HelloAgents Trip Planner 是一个面向智能旅行规划的 Agent 应用。它不是单次调用大模型生成文本，而是把用户需求理解、信息收集、工具调用、行程生成、结构化解析、自动排程、规则校验、失败修复、历史记忆和版本化存储组织成一个完整工作流。

项目的核心目标是生成可展示、可编辑、可保存、可复用的多日旅行计划。系统会根据城市、日期、天数、交通方式、住宿偏好、兴趣偏好和自由文本需求收集上下文，再由 LangGraph 编排多节点流程生成最终 `TripPlan`。

当前项目已经具备以下关键能力：

- 基于 FastAPI 提供旅行计划生成、流式生成、读取和更新接口。
- 基于 LangGraph 编排行程规划 Agent。
- 基于结构化状态管理信息收集 SOP、工具调用历史、候选数据和最终规划结果。
- 支持景点、天气、酒店、本地活动和交通时间评估等上下文能力。
- 支持用户历史偏好检索与记忆沉淀。
- 支持计划持久化、版本记录和用户编辑后的自动重排。
- 支持规划结果解析、排程、校验和修复循环。

系统可以分为六层：

- 前端展示层：负责收集用户旅行需求、展示规划结果、支持用户编辑和保存。
- API 接口层：负责请求校验、SSE 进度输出、调用 Agent、保存计划和记忆。
- Agent 编排层：基于 LangGraph 定义信息收集、规划、解析、排程、验证、修复流程。
- Tool 能力层：封装可复用的可选能力，例如 `search_local_events_tool`。
- Service 数据层：提供地图、天气、POI、活动生成、记忆、排程等底层服务。
- Repository 持久化层：负责旅行计划、版本和记忆的数据库读写。

## 4. LangGraph 工作流

当前 LangGraph 工作流主线如下：

```text
init_info_gathering
        |
        v
info_gathering_agent
        |
        +-- search_attractions_tool
        +-- query_weather_tool
        +-- search_hotels_tool
        +-- search_local_events_tool
        +-- estimate_transit_time_tool
        +-- router_warning
        +-- forced_exit_with_best_effort
        |
        v
merge_tool_result
        |
        v
info_gathering_agent
        |
        v
plan_trip
        |
        v
parse_plan
        |
        +-- parse failed and retryable -> parse_plan
        +-- parse failed and exhausted -> error_handler -> END
        |
        v
schedule_plan
        |
        v
verify_plan
        |
        +-- failed -> fix_plan -> schedule_plan
        +-- passed -> END
```

信息收集阶段由 `info_gathering_agent_node` 和 `info_gathering_router` 控制。它会优先完成必过 SOP：

- 景点信息必须收集。
- 天气信息必须收集。
- 酒店信息按请求条件决定是否必需。
- 交通评估按候选风险和固定时间活动决定是否必需。
- 本地活动是可选增强，不阻塞进入规划。

## 5. State 设计

运行态核心类型定义在 `backend/app/agents/graph_state.py`。当前状态设计已经从早期简单字段扩展为结构化运行态。

核心字段包括：

- `request`：原始旅行请求。
- `inferred_preferences`：从记忆中总结出的偏好。
- `sop_required`：当前请求需要完成哪些信息收集步骤。
- `sop_completed`：各必过步骤完成状态。
- `gathered_context`：规划前收集到的结构化上下文。
- `tool_call_history`：工具调用历史和摘要。
- `candidate_filter_notes`：候选过滤说明，主要供 planner 理解交通筛选结果。
- `ready_for_planning`：是否允许进入规划。
- `final_plan_raw` / `final_plan`：模型原始输出和结构化最终计划。
- `violations`：校验阶段发现的问题。

`gathered_context` 当前承载：

- `attractions`
- `weather`
- `hotels`
- `local_events`
- `transit_evidence`

这个设计让图层可以持续合并工具结果，也方便 planner 只读取经过整理的上下文。

## 6. Capability 封装模式

`search_local_events` 是当前项目内第一个完整落地的 capability 模板。它采用三层职责：

```text
Graph Adapter
search_local_events_node
        |
        v
Capability Registry
get_capability_tool("search_local_events_tool")
        |
        v
Tool Wrapper
@tool + SearchLocalEventsInput
        |
        v
Service
LocalEventsService.search_local_events
```

职责边界如下：

- 图层负责 orchestration：决定是否触发、组装输入、调用 tool、写回 state。
- Tool 层负责能力封装：定义 Pydantic 输入 Schema、提供 docstring 使用说明、调用 Service、输出 `text/items/warning`。
- Service 层负责数据供给：生成或检索候选、做最小标准化、做基础冲突判断。

这个模式适合后续接入更多“按需触发、可复用、输入输出稳定”的能力。基础必过能力不需要强行纳入 capability registry。

## 7. Local Events 能力

本地活动能力的定位是“惊喜感创造工具”，不是基础信息收集工具。

输入契约由 `SearchLocalEventsInput` 定义，核心字段包括：

- `city`
- `start_date`
- `end_date`
- `interest_keywords`
- `activation_reason`
- `travel_days`
- `daily_start_time`
- `daily_end_time`

输出契约为：

- `text`
- `items`
- `warning`
- `meta`

每个 local event item 保留最小核心字段：

- `name`
- `category`
- `date`
- `time_window`
- `address`
- `description`
- `interest_match_terms`
- `conflict_status`
- `conflict_reason`

Planner 侧消费规则：

- local events 是可选增强，不是主线景点。
- 优先使用 `conflict_status != "conflicting"` 的活动。
- 不要求 planner 必须使用 local events。
- 不允许 local events 覆盖基础景点主线。

## 8. Transit Phase 3 能力

交通评估能力当前主要服务于候选过滤和行程可执行性判断。

触发条件包括：

- 交通方式、候选景点或上下文显示存在跨区域风险。
- 已收集到固定时间窗口的本地活动。
- 信息收集阶段判断规划前需要交通证据。

执行结果写入：

- `gathered_context["transit_evidence"]`
- `candidate_filter_notes`
- `tool_call_history`
- `sop_completed["transit_done"]`

Planner 侧会根据 `drop_candidate` 交通证据过滤风险候选。这样 planner 不需要自己重新理解复杂交通约束，只消费已经整理过的候选和说明。

## 9. API 与持久化闭环

Trip API 当前承担完整业务闭环：

- `POST /api/trip/plan`：生成计划并保存。
- `POST /api/trip/plan/stream`：通过 SSE 返回进度和最终计划。
- `GET /api/trip/plans/{plan_id}`：读取已保存计划。
- `PUT /api/trip/plans/{plan_id}`：保存用户编辑后的计划，并触发自动重排和记忆沉淀。

持久化层主要由 repository 负责：

- `trip_repository.py`：旅行计划和版本记录。
- `memory_repository.py`：用户偏好记忆。
- `persistence_service.py`：持久化辅助能力。

数据库模型位于 `backend/app/db/models.py`，请求和响应模型位于 `backend/app/models/schemas.py`。

## 11. 后续建议

短期建议：

- 修复历史 Markdown 文档的编码显示问题，避免开发者在 PowerShell 中阅读困难。
- 将 `test_langgraph.py` 标记为手动端到端脚本，避免和快速单元测试混用。
- 为 capability registry 增加一个简短开发说明，后续新增能力时直接照模板接入。

中期建议：

- 将 `estimate_transit_time_tool` 也评估是否需要按 capability 模式封装。
- 为 planner prompt 增加更明确的候选来源说明，降低基础景点和增强活动混用风险。
- 增加一组不依赖真实 LLM/API 的端到端 smoke test，覆盖从 API request 到结构化 state 的主流程。

长期建议：

- 建立统一的能力接入规范：Service、Tool、Registry、Graph Adapter、Planner Consumption、Tests。
- 将记忆系统和兴趣关键词提取进一步结构化，增强 local events 和个性化推荐的稳定性。
- 将交通证据从文本提示逐步升级为更结构化的 planner 输入，减少模型误读风险。

## 12. 总结

当前项目已经从普通 LLM 旅行规划 Demo 发展成一个具备工程闭环的 Agent 应用。它的核心价值不只在于生成行程，而在于通过 LangGraph 把信息收集、能力调用、状态管理、规则校验、自动排程、用户记忆和持久化版本组织成可维护的系统。

从本次检查结果看，前面计划中的 phase1、local events 封装和 phase3 transit 规则已经基本执行完毕，现有自动化测试全部通过。当前主要风险不在已完成实现，而在文档编码、旧端到端脚本定位和后续能力接入规范需要继续固化。
